# Паттерны проектирования GoF (Банда четырёх)

## Порождающие паттерны (Creational)

### Паттерн: Singleton (Одиночка)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что в компании должен быть только один генеральный директор. Если бы их было несколько, это привело бы к хаосу в управлении и конфликтующим решениям.
-   **В мире IT**: Необходимость иметь единственный экземпляр какого-то объекта, доступный из любой части программы. Например, настройки приложения или подключение к базе данных должны быть одни и те же везде.

**Решение:**

-   **В повседневной жизни**: Создание строгой структуры, где определённая роль может быть только в единственном экземпляре. Как пост президента страны или капитана на корабле.
-   **В мире IT**: Паттерн гарантирует, что класс имеет только один экземпляр и предоставляет глобальную точку доступа к нему. Как если бы у вас был один главный пульт управления всей системой.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Пульт управления светофорами в городе (должен быть только один центр управления)
    -   Ключ от сейфа в банке (должен быть только один оригинальный ключ)
    -   Печать директора компании (не может быть несколько официальных печатей)
-   **В мире IT**:
    -   Системные настройки приложения
    -   Подключение к базе данных
    -   Файл конфигурации
    -   Системный принтер
    -   Логгер для записи событий

### Паттерн: Factory Method (Фабричный метод)

**Проблемы:**

-   **В повседневной жизни**: Представьте ресторан, где каждый повар должен знать все рецепты и готовить любые блюда. Это неэффективно и сложно - лучше иметь отдельных специалистов по разным кухням.
-   **В мире IT**: Необходимость создавать разные объекты, не привязываясь к конкретным классам. Как если бы вам нужно было создавать разные типы документов, не зная заранее их формат.

**Решение:**

-   **В повседневной жизни**: Создание специализированных отделов или мастерских, где каждый занимается производством определённого типа продукции. Например, в ресторане есть отдельные повара для горячих блюд, холодных закусок и десертов.
-   **В мире IT**: Определение общего интерфейса для создания объектов, но позволяет подклассам решать, какой именно класс создавать. Как фабрика, которая может производить разные модели одного продукта.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Автомобильный завод с разными конвейерами для разных моделей
    -   Пиццерия с разными способами приготовления для разных видов пицц
    -   Издательство, выпускающее разные типы книг (электронные, бумажные, аудио)
-   **В мире IT**:
    -   Создание разных типов документов в текстовом редакторе
    -   Система логирования с разными форматами вывода
    -   Создание различных элементов интерфейса в зависимости от операционной системы
    -   Генерация отчётов в разных форматах
    -   Создание подключений к разным типам баз данных

### Паттерн: Abstract Factory (Абстрактная фабрика)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы хотите обставить комнату мебелью в определённом стиле. Нужно, чтобы все предметы (стол, стулья, шкаф) соответствовали друг другу и общей концепции.
-   **В мире IT**: Необходимость создавать семейства взаимосвязанных объектов, не привязываясь к конкретным классам этих объектов. Например, создание интерфейса в едином стиле для разных операционных систем.

**Решение:**

-   **В повседневной жизни**: Создание специализированных магазинов или фабрик, которые производят полные наборы взаимосвязанных предметов. Например, мебельная фабрика, выпускающая полные гарнитуры в определённом стиле.
-   **В мире IT**: Определение интерфейса для создания семейств взаимосвязанных объектов без указания их конкретных классов. Как фабрика, которая производит не отдельные предметы, а целые наборы связанных между собой компонентов.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Производство мебельных гарнитуров в разных стилях
    -   Пошив комплектов одежды определённого бренда
    -   Создание наборов инструментов для разных профессий
    -   Комплектация автомобилей разных классов (эконом, бизнес, премиум)
-   **В мире IT**:
    -   Создание графических интерфейсов для разных операционных систем
    -   Разработка игровых уровней с разными темами оформления
    -   Формирование наборов элементов управления для разных платформ
    -   Создание комплектов документации в разных корпоративных стилях
    -   Генерация отчётов с разным оформлением для разных клиентов

### Паттерн: Builder (Строитель)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы заказываете сложное блюдо в ресторане. Его приготовление требует множества шагов, и каждый шаг должен быть выполнен в правильном порядке с учётом ваших пожеланий.
-   **В мире IT**: Необходимость создавать сложные объекты пошагово, с возможностью использовать разные способы конструирования и представления. Как если бы вам нужно было собрать компьютер из комплектующих с разными характеристиками.

**Решение:**

-   **В повседневной жизни**: Разделение процесса создания сложного продукта на отдельные шаги, каждый из которых может быть выполнен по-разному. Как сборка автомобиля на конвейере, где каждый этап чётко определён, но может иметь вариации.
-   **В мире IT**: Выделение процесса создания сложного объекта в отдельный класс (строитель), который позволяет создавать различные варианты объекта, используя один и тот же процесс строительства.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Строительство дома (фундамент → стены → крыша → отделка)
    -   Приготовление сложного блюда по рецепту
    -   Сборка автомобиля на конвейере
    -   Изготовление мебели на заказ с учётом пожеланий клиента
    -   Создание туристического маршрута с разными опциями
-   **В мире IT**:
    -   Конструктор пользовательских отчётов
    -   Создание сложных объектов конфигурации
    -   Построение SQL-запросов
    -   Формирование HTML-документов
    -   Сборка сложных объектов с множеством необязательных параметров

### Паттерн: Prototype (Прототип)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вам нужно создать копии важного документа. Перепечатывать его заново слишком долго и можно допустить ошибки, проще сделать копию на ксероксе.
-   **В мире IT**: Необходимость создавать копии объектов, не вдаваясь в подробности их реализации и не зависеть от их классов. Особенно когда создание объекта требует больших затрат ресурсов.

**Решение:**

-   **В повседневной жизни**: Использование существующего объекта как прототипа для создания новых объектов путём его копирования. Как при использовании лекала для пошива одежды или формы для отливки деталей.
-   **В мире IT**: Определение прототипного объекта, который умеет создавать копии самого себя. Вместо создания объекта "с нуля", мы клонируем существующий объект-прототип.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Копирование документов на ксероксе
    -   Использование лекал в пошиве одежды
    -   Штамповка деталей по готовой форме
    -   Создание слепков в стоматологии
    -   Размножение растений черенкованием
-   **В мире IT**:
    -   Копирование объектов в графическом редакторе
    -   Создание резервных копий состояния системы
    -   Клонирование настроек пользователя
    -   Дублирование элементов интерфейса
    -   Создание шаблонов документов на основе существующих

## Структурные паттерны (Structural)

### Паттерн: Adapter (Адаптер)

**Проблемы:**

-   **В повседневной жизни**: У вас есть американская электронная техника с вилкой под розетку американского стандарта, а розетки в доме европейского стандарта. Устройства несовместимы напрямую.
-   **В мире IT**: Необходимость заставить работать вместе классы с несовместимыми интерфейсами. Как если бы нужно было использовать старое оборудование с новым программным обеспечением.

**Решение:**

-   **В повседневной жизни**: Использование специального переходника (адаптера), который позволяет подключить вилку одного стандарта к розетке другого стандарта. Адаптер решает проблему несовместимости, не меняя сами устройства.
-   **В мире IT**: Создание специального класса-адаптера, который преобразует интерфейс одного класса в интерфейс другого, который ожидает клиент. Это позволяет работать вместе классам, которые иначе были бы несовместимы.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Переходники для электрических розеток разных стандартов
    -   Переходники для карт памяти разных форматов
    -   Переводчик между людьми, говорящими на разных языках
    -   Конвертер валют в обменном пункте
    -   Переходники для садовых шлангов разного диаметра
-   **В мире IT**:
    -   Адаптеры для работы с разными форматами данных
    -   Конвертеры форматов файлов
    -   Интеграция старых систем с новыми
    -   Адаптеры для разных API платёжных систем
    -   Преобразование данных между разными базами данных

### Паттерн: Bridge (Мост)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы производите машины разных типов (седан, хэтчбек, внедорожник) и с разными двигателями (бензиновый, дизельный, электрический). Создавать отдельный класс для каждой комбинации неэффективно.
-   **В мире IT**: Необходимость разделить большой класс или множество связанных классов на две отдельные иерархии — абстракцию и реализацию, которые можно развивать независимо.

**Решение:**

-   **В повседневной жизни**: Разделение одной большой системы на независимые части, которые могут изменяться самостоятельно. Как в производстве автомобилей: отдельно разрабатываются кузова и двигатели, которые потом можно комбинировать.
-   **В мире IT**: Разделение одной или нескольких иерархий классов на две части: абстракцию (интерфейс) и реализацию (конкретное воплощение). Это позволяет изменять их независимо друг от друга.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Производство автомобилей (разные кузова + разные двигатели)
    -   Создание мебели (разные стили + разные материалы)
    -   Производство одежды (разные фасоны + разные ткани)
    -   Строительство домов (разные проекты + разные материалы)
    -   Приготовление кофе (разные сорта + разные способы приготовления)
-   **В мире IT**:
    -   Графический редактор с разными фигурами и стилями рисования
    -   Система отображения с разными виджетами для разных платформ
    -   Драйверы устройств для разных операционных систем
    -   Система уведомлений с разными типами сообщений и способами доставки
    -   Игровой движок с разными персонажами и поведением

### Паттерн: Composite (Компоновщик)

**Проблемы:**

-   **В повседневной жизни**: Представьте организационную структуру компании, где есть отделы, подотделы и отдельные сотрудники. Нужно управлять ими единообразно, независимо от их уровня в иерархии.
-   **В мире IT**: Необходимость работать с древовидными структурами объектов, где простые и составные объекты должны обрабатываться одинаково. Например, работа с файловой системой, где папки могут содержать как файлы, так и другие папки.

**Решение:**

-   **В повседневной жизни**: Создание иерархической структуры, где каждый элемент (независимо от того, простой он или составной) имеет одинаковый интерфейс управления. Как в армии, где команды одинаково передаются как отдельным солдатам, так и целым подразделениям.
-   **В мире IT**: Объединение объектов в древовидные структуры для представления иерархий "часть-целое". Компоновщик позволяет клиентам обращаться к отдельным объектам и к группам объектов одинаково.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Организационная структура компании
    -   Военная иерархия (солдат → отделение → взвод → рота)
    -   Меню ресторана (блюдо → раздел меню → всё меню)
    -   Сборка компьютера (компонент → подсистема → целый компьютер)
    -   Структура документа (слово → абзац → глава → книга)
-   **В мире IT**:
    -   Файловая система (файл → папка → диск)
    -   Графический интерфейс (кнопка → панель → окно)
    -   HTML-документ (текст → элемент → страница)
    -   Структура XML/JSON документов
    -   Система компонентов в игровом движке

### Паттерн: Decorator (Декоратор)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы хотите кастомизировать базовую модель автомобиля разными опциями (кожаный салон, панорамная крыша, улучшенная аудиосистема), но не хотите создавать отдельный класс для каждой возможной комбинации опций.
-   **В мире IT**: Необходимость динамически добавлять новые функциональные возможности объекту, не изменяя его структуру и не создавая множество подклассов для всех возможных комбинаций функций.

**Решение:**

-   **В повседневной жизни**: Вместо изменения базового объекта, мы "оборачиваем" его в дополнительные слои, каждый из которых добавляет новую функциональность. Как если бы вы одевали человека: футболка, свитер, куртка — каждый слой добавляет новые свойства.
-   **В мире IT**: Создание специальных классов-декораторов, которые "оборачивают" основной объект и добавляют ему новое поведение. При этом все декораторы имеют тот же интерфейс, что и декорируемый объект.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Кастомизация автомобиля дополнительными опциями
    -   Добавление топпингов к мороженому или пицце
    -   Украшение новогодней ёлки
    -   Модификация базового рецепта блюда дополнительными ингредиентами
    -   Добавление аксессуаров к базовому наряду
-   **В мире IT**:
    -   Добавление скроллбаров или рамок к окну
    -   Шифрование или сжатие потока данных
    -   Добавление логирования к операциям
    -   Расширение функциональности веб-сервисов
    -   Добавление новых эффектов к изображению

### Паттерн: Facade (Фасад)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы хотите организовать путешествие. Для этого нужно забронировать билеты, отель, трансфер, экскурсии. Делать всё это по отдельности сложно и утомительно.
-   **В мире IT**: Необходимость предоставить простой интерфейс к сложной системе классов, библиотеке или фреймворку. Сложная система требует знания множества классов и их взаимодействий для выполнения даже простых операций.

**Решение:**

-   **В повседневной жизни**: Обращение в туристическое агентство, которое скрывает всю сложность организации путешествия за простым интерфейсом: вы говорите куда хотите поехать, а агентство решает все детали.
-   **В мире IT**: Создание единого класса-фасада, который предоставляет простой интерфейс для работы со сложной подсистемой. Фасад скрывает сложность системы и предоставляет только те методы, которые нужны клиенту.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Туристическое агентство (скрывает сложность организации путешествия)
    -   Ресторан быстрого питания (простое меню скрывает сложность кухни)
    -   Автосервис (мастер общается с клиентом, скрывая сложность ремонта)
    -   Консьерж в отеле (единая точка доступа ко всем услугам)
    -   Секретарь в офисе (управляет коммуникацией с руководителем)
-   **В мире IT**:
    -   Библиотека для работы с базой данных
    -   Упрощённый интерфейс для сложного API
    -   Единый интерфейс для работы с разными форматами файлов
    -   Система управления устройствами умного дома
    -   Интерфейс для работы с несколькими микросервисами

### Паттерн: Flyweight (Приспособленец)

**Проблемы:**

-   **В повседневной жизни**: Представьте библиотеку, где каждая книга существует в нескольких экземплярах. Хранить полную копию каждого экземпляра неэффективно, ведь содержание книг одинаковое.
-   **В мире IT**: Необходимость эффективно работать с большим количеством мелких объектов, которые имеют общие характеристики. Например, отображение тысяч деревьев в игре, где каждое дерево имеет одинаковую текстуру.

**Решение:**

-   **В повседневной жизни**: Разделение информации на общую (содержание книги) и уникальную (номер экземпляра, пометки читателей). Общая информация хранится в одном месте и используется всеми экземплярами.
-   **В мире IT**: Разделение объекта на две части: внутреннее состояние (общие, неизменяемые данные) и внешнее состояние (уникальные для каждого объекта данные). Общие данные хранятся в разделяемых объектах-приспособленцах.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Библиотека с несколькими копиями одной книги
    -   Использование одинаковых деталей в производстве
    -   Шаблоны документов в офисе
    -   Форма и номера у игроков в спортивной команде
    -   Использование трафаретов для рисования
-   **В мире IT**:
    -   Отображение множества графических объектов с общей текстурой
    -   Кэширование часто используемых данных
    -   Пул строк в языках программирования
    -   Общие стили для множества элементов интерфейса
    -   Разделяемые шрифты в текстовом редакторе

### Паттерн: Proxy (Заместитель)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы хотите купить дом в другом городе. Ездить каждый раз для осмотра вариантов слишком затратно. Нужен кто-то, кто будет представлять ваши интересы на месте.
-   **В мире IT**: Необходимость контролировать доступ к объекту, добавлять функциональность при обращении к нему или отложить его создание до момента реального использования. Например, загрузка тяжёлого изображения только когда оно действительно нужно.

**Решение:**

-   **В повседневной жизни**: Использование посредника (риэлтора), который действует от вашего имени: проводит осмотры, ведёт переговоры, подготавливает документы. Он предоставляет тот же интерфейс взаимодействия, что и прямой покупатель.
-   **В мире IT**: Создание объекта-заместителя, который контролирует доступ к другому объекту. Заместитель имеет тот же интерфейс, что и реальный объект, поэтому для клиента их использование идентично.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Риэлтор при покупке недвижимости
    -   Доверенное лицо в юридических вопросах
    -   Банковская карта как заместитель наличных денег
    -   Секретарь, управляющий доступом к руководителю
    -   Дипломатическое представительство страны
-   **В мире IT**:
    -   Ленивая загрузка тяжёлых ресурсов
    -   Кэширование результатов запросов
    -   Контроль доступа к защищённым ресурсам
    -   Логирование обращений к объекту
    -   Умная ссылка, подсчитывающая обращения к объекту

## Поведенческие паттерны (Behavioral)

### Паттерн: Chain of Responsibility (Цепочка обязанностей)

**Проблемы:**

-   **В повседневной жизни**: Представьте обработку жалобы в крупной компании. Сначала её рассматривает служба поддержки, если они не могут решить проблему - передают менеджеру, если и он не справляется - жалоба идёт к руководителю отдела и так далее.
-   **В мире IT**: Необходимость передавать запросы последовательно через цепочку обработчиков, пока один из них не обработает запрос. Например, система аутентификации с разными уровнями проверки.

**Решение:**

-   **В повседневной жизни**: Создание цепочки ответственных лиц, где каждый либо решает проблему сам, либо передаёт её следующему в цепочке. Как в армии, где приказ идёт по цепочке командования.
-   **В мире IT**: Создание цепочки объектов-обработчиков, где каждый объект знает о следующем в цепочке и может передать ему запрос, если не может обработать его сам.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Система рассмотрения жалоб в организации
    -   Согласование документов в бюрократической системе
    -   Эскалация проблем в службе поддержки
    -   Передача мяча между игроками в команде
    -   Процесс утверждения расходов в компании
-   **В мире IT**:
    -   Система обработки событий в пользовательском интерфейсе
    -   Фильтрация спама в почтовой системе
    -   Система логирования с разными уровнями важности
    -   Процесс аутентификации и авторизации
    -   Обработка исключений в разных слоях приложения

### Паттерн: Command (Команда)

**Проблемы:**

-   **В повседневной жизни**: Представьте ресторан, где официант принимает заказы от посетителей. Он не готовит сам, а передаёт заказ на кухню в виде стандартного бланка, который содержит всю необходимую информацию.
-   **В мире IT**: Необходимость инкапсулировать запрос как объект, позволяя параметризовать клиентов разными запросами, ставить запросы в очередь, протоколировать их или отменять операции.

**Решение:**

-   **В повседневной жизни**: Создание стандартизированной формы передачи команд, где каждая команда содержит всю необходимую информацию для её выполнения. Как военный приказ, который содержит все детали операции.
-   **В мире IT**: Преобразование запроса в отдельный объект-команду, который содержит всю информацию о запросе. Это позволяет передавать команды как параметры методам, хранить их в переменных, ставить в очередь и отменять.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Заказы в ресторане
    -   Очередь в банке с талонами
    -   Список задач для персонального ассистента
    -   Пульт дистанционного управления
    -   Рецепты в кулинарной книге
-   **В мире IT**:
    -   Операции отмены/повтора в редакторе
    -   Очередь заданий для выполнения
    -   Макросы в текстовых редакторах
    -   Планировщик задач
    -   Транзакции в базе данных

### Паттерн: Interpreter (Интерпретатор)

**Проблемы:**

-   **В повседневной жизни**: Представьте переводчика, который помогает людям, говорящим на разных языках, понять друг друга. Он должен знать правила обоих языков и уметь преобразовывать выражения из одного языка в другой.
-   **В мире IT**: Необходимость определить грамматику простого языка, представить предложения на этом языке и интерпретировать их. Например, парсинг математических выражений или создание простого языка запросов.

**Решение:**

-   **В повседневной жизни**: Создание системы правил для перевода выражений из одного формата в другой. Как нотная грамота, которая позволяет музыкантам из разных стран понимать и исполнять одну и ту же музыку.
-   **В мире IT**: Определение грамматики языка и создание интерпретатора, который может разбирать и выполнять команды этого языка. Каждое правило грамматики представляется отдельным классом.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Перевод текстов между языками
    -   Чтение нотной записи музыкантами
    -   Расшифровка азбуки Морзе
    -   Интерпретация дорожных знаков
    -   Перевод жестового языка
-   **В мире IT**:
    -   Парсинг математических выражений
    -   Обработка регулярных выражений
    -   Интерпретация SQL-запросов
    -   Разбор форматированного текста
    -   Создание предметно-ориентированных языков

### Паттерн: Iterator (Итератор)

**Проблемы:**

-   **В повседневной жизни**: Представьте библиотекаря, которому нужно просмотреть все книги на полке, не зная заранее, как они там организованы (в один ряд, в несколько рядов, стопками). Ему нужен способ последовательного доступа ко всем книгам.
-   **В мире IT**: Необходимость обеспечить способ последовательного доступа ко всем элементам составного объекта, не раскрывая его внутреннего устройства. Например, перебор элементов коллекции, независимо от того, как они хранятся.

**Решение:**

-   **В повседневной жизни**: Создание специального помощника, который знает, как обходить все элементы в определённой структуре. Как экскурсовод в музее, который знает маршрут и ведёт группу от экспоната к экспонату.
-   **В мире IT**: Выделение поведения обхода коллекции в отдельный объект-итератор. Это позволяет использовать разные способы обхода одной и той же коллекции и даже обходить несколько коллекций одновременно.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Каталог в библиотеке
    -   Меню в ресторане
    -   Обход экспонатов в музее
    -   Просмотр каналов на телевизоре
    -   Перелистывание страниц в книге
-   **В мире IT**:
    -   Обход элементов массива или списка
    -   Навигация по элементам дерева
    -   Просмотр записей в базе данных
    -   Перебор результатов поиска
    -   Потоковая обработка данных

### Паттерн: Mediator (Посредник)

**Проблемы:**

-   **В повседневной жизни**: Представьте диспетчерскую башню в аэропорту. Если бы пилоты координировали свои действия напрямую друг с другом, это привело бы к хаосу. Вместо этого они общаются через диспетчера.
-   **В мире IT**: Необходимость уменьшить связанность множества классов между собой, централизовав сложное взаимодействие и координацию между ними через посредника.

**Решение:**

-   **В повседневной жизни**: Создание центрального координатора, через которого проходит всё общение между участниками. Как модератор на совещании, который управляет порядком выступлений и обменом информацией.
-   **В мире IT**: Создание объекта-посредника, который инкапсулирует взаимодействие множества объектов. Объекты больше не общаются напрямую друг с другом, а только через посредника.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Диспетчерская служба аэропорта
    -   Модератор на конференции
    -   Риэлтор при продаже недвижимости
    -   Биржевой брокер
    -   Координатор в команде
-   **В мире IT**:
    -   Диалоговое окно с множеством элементов управления
    -   Чат-комната, где сообщения проходят через сервер
    -   Координация микросервисов через брокер сообщений
    -   Управление взаимодействием компонентов GUI
    -   Координация потоков в многопоточном приложении

### Паттерн: Memento (Снимок)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вы играете в сложную компьютерную игру и хотите сохранить своё текущее состояние, чтобы в случае неудачи вернуться к этому моменту, не начиная игру заново.
-   **В мире IT**: Необходимость сохранить внутреннее состояние объекта, чтобы иметь возможность восстановить его позже, не нарушая инкапсуляцию. Например, реализация операции "отмена" в редакторе.

**Решение:**

-   **В повседневной жизни**: Создание "контрольных точек" или "сохранений", которые содержат всю необходимую информацию для восстановления предыдущего состояния. Как фотография момента, которая позволяет воссоздать сцену позже.
-   **В мире IT**: Создание объекта-снимка, который сохраняет внутреннее состояние другого объекта, позволяя позже восстановить его в этом состоянии. При этом детали сохранения состояния остаются скрытыми от других объектов.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Сохранения в видеоиграх
    -   Резервные копии документов
    -   Закладки в книге
    -   Фотографии как память о моменте
    -   Черновики документов
-   **В мире IT**:
    -   Функция отмены в текстовых редакторах
    -   Сохранение состояния игры
    -   Контрольные точки в длительных вычислениях
    -   История изменений в системе контроля версий
    -   Восстановление состояния после сбоя

### Паттерн: Observer (Наблюдатель)

**Проблемы:**

-   **В повседневной жизни**: Представьте подписку на новостной канал. Вы хотите автоматически получать новости, как только они появляются, не проверяя постоянно сайт вручную.
-   **В мире IT**: Необходимость установить зависимость "один-ко-многим" между объектами так, чтобы при изменении состояния одного объекта все зависящие от него объекты автоматически уведомлялись и обновлялись.

**Решение:**

-   **В повседневной жизни**: Создание системы подписки, где подписчики автоматически получают уведомления об изменениях или новых событиях. Как подписка на YouTube-канал, где вы получаете уведомления о новых видео.
-   **В мире IT**: Определение механизма подписки, позволяющего одним объектам следить и реагировать на события, происходящие в других объектах. Издатель содержит список подписчиков и оповещает их об изменениях.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Подписка на новостную рассылку
    -   Подписка на обновления в социальных сетях
    -   Система оповещения о чрезвычайных ситуациях
    -   Отслеживание статуса доставки
    -   Мониторинг биржевых котировок
-   **В мире IT**:
    -   Обновление пользовательского интерфейса при изменении данных
    -   Система событий в браузере (клики, нажатия клавиш)
    -   Обработка изменений в базе данных
    -   Система логирования действий пользователя
    -   Рассылка уведомлений в мессенджере

### Паттерн: State (Состояние)

**Проблемы:**

-   **В повседневной жизни**: Представьте торговый автомат, поведение которого зависит от его текущего состояния. Когда в нём есть товары и деньги для сдачи, он работает нормально. Когда товары или деньги заканчиваются, он должен вести себя иначе.
-   **В мире IT**: Необходимость изменять поведение объекта в зависимости от его внутреннего состояния, при этом избегая больших условных операторов, проверяющих текущее состояние объекта.

**Решение:**

-   **В повседневной жизни**: Определение чётких правил поведения для каждого возможного состояния объекта. Как светофор, который имеет три состояния, и для каждого определено своё поведение.
-   **В мире IT**: Выделение состояний объекта в отдельные классы и делегирование им определённого поведения. Объект делегирует выполнение задач объекту состояния вместо того, чтобы иметь множество условных операторов.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Светофор с разными режимами работы
    -   Торговый автомат с разными состояниями
    -   Стиральная машина с разными режимами стирки
    -   Телефон (режимы звонка, вибрации, без звука)
    -   Система охраны (включена, выключена, тревога)
-   **В мире IT**:
    -   Мультимедиа плеер (воспроизведение, пауза, стоп)
    -   Процесс заказа в интернет-магазине
    -   Документ в документообороте
    -   Подключение к сети (поиск, подключено, отключено)
    -   Игровой персонаж с разными состояниями

### Паттерн: Strategy (Стратегия)

**Проблемы:**

-   **В повседневной жизни**: Представьте, что вам нужно добраться из пункта А в пункт Б. У вас есть разные варианты: ехать на машине, лететь самолётом, плыть на корабле. Каждый способ имеет свои преимущества и недостатки.
-   **В мире IT**: Необходимость определить семейство алгоритмов, инкапсулировать каждый из них и сделать их взаимозаменяемыми. Позволяет клиенту выбирать нужный алгоритм независимо от использующих его объектов.

**Решение:**

-   **В повседневной жизни**: Выделение различных способов достижения цели в отдельные стратегии, которые можно легко заменять. Как выбор маршрута в навигаторе: пешком, на машине или на общественном транспорте.
-   **В мире IT**: Определение семейства алгоритмов, помещение каждого из них в собственный класс и создание механизма для выбора нужного алгоритма во время выполнения программы.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Выбор способа путешествия
    -   Выбор способа оплаты (наличные, карта, онлайн)
    -   Выбор тактики в спортивной игре
    -   Выбор метода обучения
    -   Выбор диеты для похудения
-   **В мире IT**:
    -   Алгоритмы сортировки данных
    -   Стратегии сжатия файлов
    -   Методы аутентификации пользователей
    -   Алгоритмы расчёта стоимости доставки
    -   Форматы экспорта данных

### Паттерн: Template Method (Шаблонный метод)

**Проблемы:**

-   **В повседневной жизни**: Представьте процесс приготовления разных видов кофе. Основные шаги похожи (вскипятить воду, добавить кофе), но детали могут отличаться (добавление молока, сахара, сиропа).
-   **В мире IT**: Необходимость определить скелет алгоритма, позволяя подклассам переопределять определённые шаги алгоритма, не меняя его структуру в целом.

**Решение:**

-   **В повседневной жизни**: Создание базового рецепта или инструкции, определяющей последовательность шагов, где некоторые шаги могут быть изменены для получения различных вариаций. Как базовый рецепт пиццы, где начинка может меняться.
-   **В мире IT**: Определение основного алгоритма в базовом классе, где некоторые шаги оставлены для реализации в подклассах. Базовый класс контролирует общую структуру алгоритма, а подклассы предоставляют конкретную реализацию отдельных шагов.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Приготовление разных видов кофе
    -   Сборка мебели по инструкции
    -   Процедура посадки самолёта
    -   Проведение медицинского осмотра
    -   Процесс обслуживания клиента в банке
-   **В мире IT**:
    -   Фреймворки для создания приложений
    -   Процесс обработки данных
    -   Этапы сборки программного продукта
    -   Шаблоны отчётов с разным содержимым
    -   Обработка платежей разными системами

### Паттерн: Visitor (Посетитель)

**Проблемы:**

-   **В повседневной жизни**: Представьте медицинский осмотр, где разные врачи-специалисты (окулист, терапевт, хирург) проводят свои специфические обследования одного и того же пациента, не меняя его состояние.
-   **В мире IT**: Необходимость добавить новую операцию для целой иерархии классов, не изменяя сами классы. Например, добавить возможность экспорта разных типов документов в различные форматы.

**Решение:**

-   **В повседневной жизни**: Вместо того чтобы добавлять разные типы проверок в класс пациента, создаются отдельные специалисты-посетители, каждый из которых знает, как работать с пациентом по своему профилю.
-   **В мире IT**: Выделение операций над объектом в отдельные классы-посетители. Это позволяет добавлять новые операции, не изменяя классы объектов, над которыми эти операции выполняются.

**Примеры применения:**

-   **В повседневной жизни**:
    -   Медицинский осмотр разными врачами
    -   Проверка документов разными инстанциями
    -   Инспекция разных отделов компании
    -   Обслуживание техники разными специалистами
    -   Оценка имущества разными экспертами
-   **В мире IT**:
    -   Операции с различными узлами AST (абстрактного синтаксического дерева)
    -   Форматирование разных типов документов
    -   Валидация сложных структур данных
    -   Сбор статистики по разным типам объектов
    -   Применение различных операций к элементам коллекции
